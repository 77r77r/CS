## HDD Scheduling

하드 스케쥴링은 접근 시간(seek time)을 최소화하는 것과 데이터 전송의 bandwidth을 최대화 시키는 것이 목표다.

seek time은 특정 실린더의 특성 세터를 찾아가는 시간을 이야기한다.
disk bandwidth은 한 번에 전송할 수 있는 용량 / 전체 시간 을 이야기한다. 섹터가 클수록 전송할 수 있는 용량이 커진다.

FIFO가 낭비되는 이동 거리가 많아 나온 것이 SCAN 스케쥴링이다. 왔다갔다 하지 말고 양방향으로 쭉 이동하면서 목적지를 다 읽어처리하는 방식이다.
양방향으로 왔다갔다 하는것보다 한쪽 방향을 정해서 그쪽으로만 쭉 동작하는 방식이 C-SCAN 방식이다. 갈때만 읽고, 되돌아 올때는 읽지 않는다. 원형 리스트처럼 실린더를 다루는방식.

## RAID: Redundant Arrays of Independent Disks

RAID는 여러 개의 디스크를 하나의 시스템처럼 사용하여 성능과 신뢰성을 높이는 디스크 구성 기술이다. 여러 디스크를 병렬로 동작시켜 데이터 읽기와 쓰기 속도를 향상시킬 수 있으며, 동시에 데이터를 여러 장치에 중복
저장하여 저장의 신뢰성을 높일 수 있다. 따라서 하나의 디스크가 고장 나더라도 데이터 손실을 방지할 수 있다.

### Redundancy: 신뢰성 향상

여러 개의 디스크로 구성된 시스템에서는 디스크 수가 많아질수록 전체 시스템에서 고장이 발생할 확률이 증가한다. 예를 들어 하나의 디스크 MTBF(Mean Time Between Failures) 가
100,000시간이라면, 100개의 디스크로 구성된 시스템에서는 MTBF가 약 1,000시간으로 감소한다.

이러한 신뢰성 문제를 해결하기 위해 중복성(Redundancy) 을 활용한다. 가장 단순한 방법은 미러링(Mirroring) 으로, 모든 디스크의 데이터를 동일하게 복제하여 저장하는 방식이다. 단, 같은 물리적
공간에서 미러링하는 것은 의미가 없으며 데이터 센터가 물리적으로 분산되어 있어야 실질적인 신뢰성을 확보할 수 있다. 신뢰성은 보장되지만 저장 비용이 두 배로 증가한다는 단점이 있다.

> \*MTBF(Mean Time Between Failures)는 한 시스템이나 장치가 고장과 고장 사이에서 평균적으로 정상 동작하는 시간이다. 신뢰도를 측정하는 가중 대중적인 기준으로, 전체 가동 시간을 발생한
> 고장 횟수로 나누어 계산한다.

### Parallelism: 성능 향상

병렬성(Parallelism) 은 여러 개의 디스크를 동시에 사용하여 성능을 향상시키는 방식이다. 이를 위해 데이터를 여러 디스크에 분산 저장하는 스트라이핑(Striping) 기법을 사용한다.
저장하는 데이터의 단위에 따라 방식이 나뉜다. 비트 단위로 분산 저장하면 Bit-level Striping, 블록 단위로 분산 저장하면 Block-level Striping 방식이다.
여러 디스크가 동시에 데이터를 처리하므로 전송 속도를 크게 향상시킬 수 있다.
단, 스트라이핑은 성능은 높지만 하나의 디스크라도 고장나면 데이터 전체를 잃을 수 있어 신뢰성을 보장하지 못하는 단점이 있다.

### Parity bit

패리티 비트(Parity Bit)는 데이터 전송이나 저장 과정에서 오류를 검출하기 위해 추가하는 1비트다. 전체 1의 개수가 짝수가 되도록 설정하는 Even Parity(짝수 패리티) 와 홀수가 되도록 설정하는
Odd Parity(홀수 패리티) 방식이 있다. 규칙과 1의 개수가 맞지 않으면 오류가 발생한 것으로 판단한다. 단 1비트만 추가하여 오류를 검출할 수 있다는 장점이 있지만, 2개의 비트가 동시에 깨진 경우 오류를
검출하지 못하는 한계가 있다. 이를 보완하기 위해 Checksum 과 CRC(Cyclic Redundancy Check) 로 확장될 수 있다.

### RAID (Redundant Array of Independent Disks)

데이터의 오류 검출 및 복구 가능 여부, 그리고 비용에 따라 RAID 레벨이 달라진다.

**RAID 0** 은 스트라이핑만 적용한 가장 기본적인 방식이다. 성능은 높지만 중복성이 없어 하나의 디스크가 고장나면 데이터 전체를 잃는다.

**RAID 1** 은 미러링을 적용한 방식이다. 데이터를 그대로 복제하여 저장하므로 신뢰성이 높지만 저장 비용이 2배로 증가한다.

**RAID 4** 는 데이터를 저장하는 디스크와 패리티 비트를 저장하는 디스크를 분리하여 운영하는 방식이다.

**RAID 5** 는 패리티 비트를 별도의 디스크에 집중시키지 않고 각 디스크에 분산하여 저장하는 방식이다. RAID 4의 패리티 디스크 병목 문제를 해결한다.

**RAID 6** 은 RAID 5에서 패리티를 하나 더 추가(P+Q)하여 두 개의 디스크가 동시에 고장나도 복구할 수 있는 방식이다.

**RAID 7** 은 P와 Q 데이터를 다차원 배열로 별도 관리하는 방식이다.

실제 시스템에서는 RAID 레벨을 단독으로 사용하기도 하지만, RAID 0+1 처럼 여러 레벨을 조합하여 성능과 신뢰성을 동시에 확보하는 방식도 많이 사용된다. 시스템 요구사항에 맞는 RAID 레벨을 선택하는 것이 중요하다.


## I/O System

컴퓨터의 주요 작업은 입출력(I/O) 과 연산 으로 나눌 수 있다. 웹 브라우징, 파일 편집, 영상 시청, 게임 등 실제 많은 작업이 I/O 중심으로 이루어진다. 운영체제는 다양한 I/O 장치와 I/O 작업을
관리하고 제어하는 역할을 담당한다.
각 I/O 장치는 디바이스 컨트롤러(Device Controller) 를 통해 PCI 버스 에 연결되며, CPU는 이 버스를 통해 각 장치에 명령을 전달한다.

### Memory-Mapped I/O

프로세서는 입출력(I/O)을 수행하기 위해 레지스터를 통해 컨트롤러에 명령과 데이터를 전달한다.
I/O 컨트롤러에는 data-in, data-out, status, control 레지스터가 있으며, CPU는 일반적인 메모리 접근 명령을 사용하여 이러한 레지스터를 읽거나 씀으로써 I/O 장치를 제어한다.

- data-in register: 호스트(CPU)가 입력 데이터를 얻기 위해 읽는 레지스터
- data-out register: 호스트(CPU가) 출력 데이터를 장치로 보내기 위해 쓰는 레지스터
- status register: 장치의 현재 상태를 비트 단위로 표기(Read-only)
- control register: 호스트(CPU)가 장치에게 명령을 시작하거나 모드를 변경하기 위해 사용(Write-only)

Memory-Mapped I/O는 I/O 장치의 제어 레지스터를 프로세서의 메모리 주소 공간에 매핑하여 접근하는 방식이다.
특정 메모리 주소에 어떤 I/O 장치가 연결되어 있는지 매핑해두고, CPU가 해당 주소에 명령을 전달하면 장치 컨트롤러가 이를 받아 처리하는 구조다.
CPU는 별도의 I/O 명령어 없이 일반적인 메모리 읽기/쓰기 명령만으로 장치를 제어할 수 있다.

### Three types of I/O

입출력에 3가지 타입이 있다.  
Polling(폴링) 은 CPU가 상태 레지스터를 반복적으로 읽으며 busy 비트가 해제될 때까지 대기하는 방식이다. 구현이 단순하지만 CPU가 I/O 완료를 기다리는 동안 다른 작업을 하지 못하는
Busy-Waiting 문제가 발생한다.  
Interrupt(인터럽트) 는 CPU가 I/O 완료를 기다리지 않고 다른 작업을 수행하다가, 장치가 준비되면 인터럽트 신호를 통해 알려주는 방식이다. CPU는 인터럽트 요청 라인(Interrupt-Request
Line)을 통해 인터럽트를 감지하고, 감지되면 ISR(Interrupt Service Routine)로 점프하여 처리한다. 각 ISR의 주소는 인터럽트 벡터 테이블(Interrupt Vector Table) 에
저장되어 있다.    
DMA(Direct Memory Access) 는 CPU를 거치지 않고 I/O 장치가 메모리에 직접 데이터를 전송하는 방식이다. 바이트 단위로 CPU가 직접 처리하는 Programmed I/O(load, store)
의 비효율을 개선하기 위해 사용하며, 대용량 데이터 전송에 특히 유용하다.

#### Interrupt-driven I/O cycle

```text
1. Device driver가 I/O 요청을 시작
CPU에서 장치 드라이버가 I/O 작업을 요청한다.

2. I/O controller가 I/O 작업을 수행
컨트롤러가 실제 장치와 데이터를 주고받기 시작한다.

3. I/O 작업 완료 또는 오류 발생
입력 준비, 출력 완료, 또는 오류가 발생하면 인터럽트 신호를 생성한다.

4. CPU가 인터럽트를 감지
CPU는 명령 실행 사이에서 인터럽트를 확인하고, 인터럽트가 발생하면 interrupt handler로 제어를 넘긴다.

5. Interrupt handler 실행
인터럽트 서비스 루틴(ISR)이 실행되어 데이터 처리나 장치 상태 처리를 수행한다.

6. 인터럽트 처리 종료
ISR 실행이 끝나면 인터럽트에서 복귀(return from interrupt) 한다.

7. CPU가 원래 작업을 다시 수행
인터럽트 때문에 중단되었던 기존 프로그램 실행을 계속한다.
```

### Blocking I/O .vs. Non-blocking I/O

**Blocking I/O**는 I/O 작업이 완료될 때까지 스레드가 실행을 멈추고 Running Queue에서 Waiting Queue로 이동하는 방식이다. 작업이 완료되면 다시 Running Queue로 돌아와
실행을 재개한다.

**Non-blocking I/O**는 스레드의 실행을 중단하지 않고 현재 사용 가능한 데이터만 즉시 반환하면서 계속 실행되는 방식이다. 요청한 데이터의 전부, 일부, 또는 아무 데이터도 반환하지 않을 수 있다.

**Asynchronous I/O(비동기 I/O)** 는 I/O 요청 이후에도 스레드가 자신의 코드를 계속 실행하는 방식이다. 전체 데이터 전송을 요청하고 해당 작업은 백그라운드에서 수행되어 미래의 어느 시점에 완료
신호를 받는다.

> Non-blocking은 "지금 줄 수 있는 것만 바로 반환"하는 방식이고, Asynchronous는 "전부 처리할 테니 나중에 알려줄게"하는 방식이라는 점에서 차이가 있다.

## File System

파일 시스템은 운영체제와 사용자의 데이터 및 프로그램을 온라인 저장장치에 저장하고 접근할 수 있도록 하는 메커니즘이다. 파일 시스템은 두 가지 부분으로 구성된다. 관련 데이터를 저장하는 파일들의 집합과, 시스템 내
모든 파일을 체계적으로 정리하는 디렉토리 구조(Directory Structure) 이다.

### Access Methods (접근 방식)

**Sequential Access(순차 접근)** 는 파일의 데이터를 처음부터 순서대로 하나씩 처리하는 방식이다. 테이프와 같은 구조로, 앞에서부터 순서대로만 읽고 쓸 수 있다.

**Direct Access(직접 접근)** 는 고정 길이의 논리 레코드로 구성된 파일에서 순서에 상관없이 원하는 레코드에 빠르게 읽고 쓸 수 있는 방식이다. 상대 접근(Relative Access)이라고도 하며,
디스크처럼 임의의 위치에 바로 접근할 수 있다.
이를 구현한 것이 RAF(Random Access File) 로, Java에서는 RandomAccessFile 클래스를 통해 파일의 특정 위치로 바로 이동하여 읽고 쓸 수 있다.

### Directory Structure (디렉토리 구조)

디렉토리는 파일 이름을 해당 파일의 FCB(File Control Block) 으로 변환해주는 심볼 테이블로 볼 수 있다. 디렉토리 구조를 구성하는 방식은 네 가지가 있다.

Single-Level Directory(단일 수준 디렉토리) 는 모든 파일이 하나의 디렉토리에 존재하는 가장 단순한 구조다.

Two-Level Directory(2단계 디렉토리) 는 사용자마다 별도의 디렉토리를 두는 구조로, 사용자 간 파일 이름 충돌을 방지할 수 있다.

Tree-Structured Directories(트리 구조 디렉토리) 는 디렉토리를 트리 형태로 계층적으로 구성하는 방식이다. 현대 운영체제에서 가장 일반적으로 사용되는 구조다.

Acyclic-Graph Directories(비순환 그래프 디렉토리) 는 트리 구조를 확장하여 하나의 파일이나 디렉토리를 여러 디렉토리에서 공유할 수 있도록 한 구조다. 순환(Cycle)이 없는 그래프 형태로
구성된다.

### Layered File System (계층적 파일 시스템)

파일 시스템은 여러 계층으로 구성된다.  
상위 계층일수록 추상화 수준이 높고, 하위 계층일수록 실제 하드웨어에 가깝다. 각 계층은 바로 아래 계층의 기능만 사용하므로 계층 간 의존성을 낮출 수 있다.

```text
Application Programs      // 사용자 애플리케이션
       ↓
Logical File System       // 파일 메타데이터, 디렉토리 구조 관리
       ↓
File-Organization Module  // 논리 블록을 물리 블록으로 변환
       ↓
Basic File System         // 실제 블록 단위 읽기/쓰기 명령 처리
       ↓
I/O Control               // 디바이스 드라이버, 인터럽트 처리
       ↓
Devices                   // 실제 저장 장치 (HDD, SSD 등)
```

### Allocation Method (할당 방식)

파일 시스템 구현의 핵심 문제는 저장 공간을 효율적으로 활용하면서도 파일에 빠르게 접근할 수 있도록 공간을 할당하는 것이다. 주요 할당 방식은 세 가지다.

**Contiguous Allocation(연속 할당)** 은 파일을 연속된 블록에 저장하는 방식이다. 구현이 단순하고 순차 접근과 직접 접근 모두 빠르다는 장점이 있지만, 외부 단편화가 발생하고 이를 해결하기 위한
압축(Compaction) 작업이 필요하다는 단점이 있다.

**Linked Allocation(연결 할당)** 은 파일의 블록들을 연결 리스트 형태로 저장하는 방식이다. 블록들이 장치 어디에나 흩어져 있어도 되므로 외부 단편화와 압축 문제가 없다. 그러나 블록을 순서대로
따라가야 하므로 직접 접근(Direct Access)이 느리다는 단점이 있다.  
이를 개선한 방식이 **FAT(File Allocation Table)** 로, 각 볼륨의 시작 부분에 테이블을 별도로 두고 블록 간 연결 정보를 한곳에 집중시킨다. 테이블을 캐시에 올려두면 디스크를 직접 탐색하지
않고도 원하는 블록의 위치를 빠르게 찾을 수 있다.

**Indexed Allocation(인덱스 할당)** 은 FAT 없는 연결 할당의 문제, 즉 블록 포인터가 각 블록에 흩어져 있는 문제를 해결하기 위해 모든 포인터를 인덱스 블록(Index Block) 하나에
모아두는 방식이다. 각 파일은 자신만의 인덱스 블록을 가지며, 인덱스 블록의 i번째 항목이 파일의 i번째 블록을 가리킨다. 블록 포인터가 흩어져 있는 연결 할당의 문제를 해결하면서도 원하는 블록에 빠르게 직접 접근할
수 있다.

> 디지털 포렌식에서는 인덱스 블록을 활용하여 삭제된 파일의 흔적을 추적하거나 파편화된 데이터를 재구성하는 데 활용된다.

### ree-Space Management (빈 공간 관리)

사용 가능한 디스크 공간을 추적하기 위해 시스템은 Free-Space List 를 유지한다. 새로운 파일이 생성되거나 기존 파일이 삭제될 때 이 목록을 갱신하여 어느 블록이 비어있는지 관리한다.
