## 사전 지식

#### 하드웨어 계층 구조

메모리는 독립적으로 존재하는 것이 아니라 속도와 용량에 따라 계층을 나눈다. CPU 내부의 가장 빠른 저장소인 레지스터(Register), CPU와 RAM 사이의 속도 차이를 줄여주는 캐시 메모리(Cache,
L1/L2/L3), 그리고 HDD·SSD 같은 비휘발성 보조 기억 장치(Storage)로 구성된다.

#### 주소값이란?

메모리는 각기 고유한 주소를 가진 바이트 단위의 큰 배열로 구성되어 있다. CPU는 프로그램 카운터(PC)에 저장된 메모리 주소를 사용하여 다음에 실행할 명령어를 가져오고, Load와 Store 동작을 통해 메모리에
접근한다.

주소는 두 종류로 나뉜다. 물리 주소(Physical Address) 는 실제 RAM 칩에 매겨진 하드웨어 주소이고, 논리 주소(Logical Address) 는 CPU가 생성하는 가상의 주소로 실제 물리 주소와는
독립적이다. 이 두 주소를 실시간으로 변환해주는 하드웨어 장치가 MMU(Memory Management Unit) 이다.

#### 프로세스와 메모리

프로세스는 메모리에 로드되어 실행 중인 프로그램이다. 프로세스는 메모리를 네 가지 영역으로 나누어 사용한다. 실행할 명령어를 저장하는 Code 영역, 전역·정적 변수를 저장하는 Data 영역, 동적 할당 데이터를
저장하는 Heap 영역, 지역 변수와 함수 호출 정보를 저장하는 Stack 영역이다.

---

프로세스는 메인 메모리에 로드되어 실행 중인 프로그램이다. 이 중 사용자 모드(User Mode)에서 실행되는 프로세스를 특별히 유저 프로세스(User Process) 라고 한다.
메모리는 각기 고유한 주소를 가진 바이트 단위의 큰 배열로 구성되어 있다. 주소는 메모리 안의 한 칸(1바이트)을 가리키며, CPU는 프로그램 카운터(PC)에 저장된 주소를 사용해 다음에 실행할 명령어를 가져온다.
이후 Load와 Store 동작을 통해 메모리에 접근하며 명령어를 실행한다.

> 메모리의 주소공간(Memory Space)에 프로세스를 어떻게 저장하고 관리하는지를 중점적으로 살펴보자.

## Memory Space

각 프로세스가 서로 독립된 메모리 공간을 갖는 것이 멀티 프로그래밍의 기본이다. 공유 메모리가 아닌 한 자신에게 할당된 주소에만 접근할 수 있으며, Base Register와 Limit Register를 이용해
허용된 범위를 관리한다. 접근하려는 주소가 허용된 범위를 벗어나면 Segmentation Fault가 발생한다. 세그멘테이션 폴트는 프로세스가 허용되지 않은 메모리 영역에 접근했을 때 발생하는 오류이다.

> 배열 범위 초과 접근이 항상 Segmentation Fault를 발생시키는 것은 아니다.  
> 배열은 연속된 메모리 주소에 배치되므로 벗어난 범위가 프로세스에 허용된 주소 내라면 오류가 발생하지 않는다.

## Address Binding

프로그램은 디스크에 실행 파일(binary)로 존재하다가 메모리에 적재되는 순간 프로세스가 된다. 소스 코드에서 변수나 함수는 기호적(symbolic) 주소로 표현되고, 컴파일러가 이를 재배치 가능한 주소(
relocatable address)로 변환한다. 이후 링커 또는 로더가 재배치 주소를 실제 메모리 주소인 절대 주소(absolute address)로 변환한다.

링커(Linker)는 컴파일러가 만든 여러 오브젝트 파일(.o)들을 하나의 실행 파일로 합치는 도구로, 작성한 코드와 외부 라이브러리(printf 등)를 연결해서 실행 가능한 파일로 만들어주는 역할을 한다.  
로더(Loader)는 링커가 만든 실행 파일을 실제 메모리에 올려주는 도구로, 이 시점에 논리 주소가 실제 물리 주소로 확정된다.

```text
소스코드 → [컴파일러] → 오브젝트 파일
오브젝트 파일 → [링커] → 실행 파일
실행 파일 → [로더] → 메모리에 적재 → 프로세스
```

현대 OS는 가상 메모리를 사용하기 때문에 두 프로그램이 동일한 논리 주소를 가져도 실제 물리 주소는 다를 수 있다. C/C++은 메모리 주소를 직접 다룰 수 있는 반면, Java·Python은 주소가 아닌 객체
참조로 메모리를 다루며 GC(Garbage Collector)가 자동으로 메모리를 관리한다.

> **GC(Garbage Collector)**란 더 이상 참조되지 않는 객체를 추적하여 메모리를 자동으로 해제하는 것이다. 필요하면 단편화 방지를 위해 살아있는 객체들을 모아 정리하기도 한다.

## Logical vs Physical

논리 주소(Logical Address)는 CPU가 생성하는 주소로, 사용자 프로세스가 접근하고자 하는 주소다. 실제 물리 주소와는 독립적으로 존재한다. 물리 주소(Physical Address)는 실제 메모리
하드웨어에 매겨진 주소로, 메모리 주소 레지스터(MAR)에 적재되는 실제 주소다.
논리 주소 공간은 하나의 사용자 프로그램이 생성할 수 있는 모든 논리 주소의 집합이고, 물리 주소 공간은 그 논리 주소들이 변환되어 대응되는 실제 물리 주소들의 집합이다. 사용자 프로그램은 항상 논리 주소만 다루고
실제 물리 주소는 직접 알 수 없으며, 이 두 주소를 연결해주는 것이 MMU다.

## Memory Management Unit(MMU)

```text
CPU → (논리 주소) → MMU → (물리 주소) → Physical Memory
```

MMU는 논리 주소를 물리 주소로 변환하는 하드웨어 장치이다. Relocation Register를 가지고 있으며 이것이 Base Register 역할을 한다.

## Dynamic Loading 과 Dynamic Linking

**Dynamic Loading(동적 로딩)** 은 프로그램 전체를 메모리에 올리지 않고, 필요한 루틴이 호출될 때만 메모리에 적재하는 방식이다. 사용되지 않는 코드는 메모리에 올라오지 않으므로 메모리 사용 효율을
높일 수
있다.

**Dynamic Linking(동적 링킹)** 은 실행 시점에 라이브러리와 연결하는 방식이다. 시스템 라이브러리를 컴파일 시점에 포함시키는 정적 링킹과 달리, 동적 링킹은 여러 프로그램이 하나의 DLL을 공유할 수
있어 메모리를 절약할 수 있다.  
**DLL(Dynamically Linked Libraries)** 이란 공유 라이브러리로, Windows는 .dll, Linux/Unix는 .so, macOS는 .dylib 파일을 사용하며,
printf(), fork() 같은 시스템 함수가 대표적인 예다.

## Contiguous Memory Allocation : 연속 메모리 할당

메모리는 한정된 자원으로 효율적으로 할당되어야 한다. 운영체제 영역과 사용자 프로세스 영역으로 나뉜다. 여러 개의 사용자 프로세스가 동시에 메모리에 상주해야 하기 때문에 어떻게 할당할 것인지가 핵심 문제이다.

**연속 메모리 할당(Contiguous Memory Allocation)** 은 각 프로세스에게 하나의 연속된 메모리 블록을 통째로 주는 방식이다. 시작 주소Relocation Register)와 끝 주소(
Limit Register)로 영역을 관리하며, 프로세스가 자신이 소유하지 않은 메모리에 접근하는 것을 방지한다. 할당 방식은 고정 분할과 가변 분할로 나뉜다.

### Fixed-Partition (고정 분할)

메모리를 미리 정해진 크기의 파티션으로 나누어 각 파티션에 하나의 프로세스를 할당하는 방식이다. 구현이 단순하지만 프로세스 크기가 파티션보다 작으면 남는 공간이 낭비되는 Internal Fragmentation(내부
단편화) 문제가 발생한다.

### Variable-Partition (가변 분할)

프로세스 크기에 맞게 메모리를 동적으로 나누어 할당하는 방식이다. 내부 단편화는 없지만, 프로세스가 올라오고 나가는 과정에서 빈 공간인 Hole이 생기며 External Fragmentation(외부 단편화) 문제가
발생한다.

Dynamic Storage Allocation이란, 이 Hole에 프로세스를 배치하는 전략으로, 처음 만나는 Hole에 할당하는 **First-Fit**, 가장 작은 Hole에 할당하는 **Best-Fit**,
가장 큰 Hole에 할당하는 **Worst-Fit** 세 가지가 있다. First-Fit과 Best-Fit이 Worst-Fit보다 효율적이다.

이 방식은 구조가 단순한 대신, 빈 공간이 여러 곳에 흩어져 있으면 합산 크기는 충분해도 연속되지 않아 프로세스를 올리지 못하는 **외부 단편화**, 그리고 필요한 것보다 큰 블록을 할당받아 남는 공간이 낭비되는 *
*내부 단편화**가 발생한다.

이러한 단편화 문제를 해결하기 위해 등장한 것이 **페이징(Paging)** 과 **세그멘테이션(Segmentation)**이다.  
세그멘테이션은 메모리를 스택, 힙, 코드, 데이터 등 논리적인 단위인 세그먼트로 분할하는 방법으로, 프로세스를 통째로 올리는 방식보다 유연하지만 세그먼트 크기가 가변적이기 때문에 외부 단편화 문제가 더 심하게 발생할
수 있다.

## 페이징과 스와핑

## Paging (페이징)

프로세스의 물리 주소 공간이 연속되지 않아도 되는 메모리 관리 기법이다. 외부 단편화가 발생하지 않고, 따라서 이를 해결하기 위한 압축(Compaction) 작업이 필요하지 않다. 운영체제와 하드웨어의 협력으로
구현된다.

기본 방식은 물리 메모리를 고정된 크기의 블록인 **프레임(Frame)** 으로 나누고, 논리 메모리를 동일한 크기의 블록인 **페이지(Page)** 로 나누는 것이다. 프로그램이 페이지 단위로 나뉘기 때문에 물리
메모리에 순서대로 올릴 필요가 없으며, 운영체제가 페이지와 프레임의 매핑을 담당하여 논리 주소 공간과 물리 주소 공간을 완전히 분리한다.

어떤 페이지인지를 나타내는 **페이지 번호(page number, p)** 와 해당 페이지 내에서의 위치를 나타내는 **페이지 오프셋(page offset, d)** 으로 나눠진다. 물리 주소를 직접 사용할 필요
없이 p페이지의 d번 주소에 접근할 수 있다. 페이지 넘버의 개수와 페이지 오프셋의 개수로 논리 주소의 크기가 결정된다.

### Page Table

프로세스 별로 페이지 개수가 다르기 때문에 별도의 테이블(Page Table)에 페이지 넘버가 관리되어야 한다.

논리 주소를 물리 주소로 변환하는 과정은 다음과 같다. 먼저 논리 주소에서 페이지 번호(p)를 추출해 페이지 테이블의 인덱스로 사용한다. 페이지 테이블에서 해당 페이지 번호에 대응하는 프레임 번호(f)를 가져온다.
마지막으로 논리 주소의 페이지 번호(p)를 프레임 번호(f)로 교체하면 최종 물리 주소가 완성된다.

```text
CPU -> 논리 주소(p, d) -> Page Table -> 물리 주소(f, d) -> 물리 메모리
```

외부 단편화는 발생하지 않지만, 내부 단편화가 발생한다.

### 페이지의 크기

페이지의 크기는 하드웨어의 의해 결정되며, 일반적으로 4KB에서 1GB 사이의 2의 거듭제곱으로 결정된다.  
논리 주소 공간의 크기가 2^m이고 페이지 크기가 2^n일 때, 논리 주소의 상위 m-n 비트가 페이지 번호(p)를, 하위 n 비트가 페이지 오프셋(d)을 나타낸다.

- m = 논리 주소 전체 비트 수
- n = 페이지 크기(오프셋)를 표현하는 비트 수
- m-n = 페이지 번호를 표현하는 비트 수

예를 들어, 페이지 크기가 2^2 = 4바이트라면 n=2으로 오프셋이 2비트이다.
전체 페이지 개수가 2^10개라면 페이지 번호를 표현하는 데 10비트가 필요하므로 m-n=10, 즉 m=12이다.

```text
논리 주소 전체 = 12비트 (m=12)
페이지 번호   = 10비트 (m-n = 12-2 = 10)
오프셋        =  2비트 (n=2)
```

새로운 프로세스가 도착하면 해당 프로세스의 크기를 페이지 단위로 환산하여 필요한 프레임 수를 계산한다. 이후 빈 프레임 목록에서 필요한 만큼 프레임을 할당받아 프로세스의 각 페이지를 적재한다.

##                

CPU 스케줄러가 새로운 프로세스를 실행하기위해 문맥교환이 발생하면 페이지 테이블이 그대로 있을지 모르기 때문에 포함되어야 한다. 각 프로세스의 페이지 테이블 포인터는 PCB(Process Control Block)
에 다른 레지스터 값들과 함께 저장된다.

페이지 테이블의 크기가 큰 경우 관리하기 힘들기 때문에 **PTBR(Page-Table Base Register)** 을 통해 관리된다. 페이지 테이블은 메모리에 적재하고 PTBR을 이용하여 페이지 테이블의 시작
주소를 가르킨다. 이 경우 컨텍스트 스위치 시 PTBR 값만 변경하면 되므로 컨텍스트 스위치는 빠르지만, 메모리 접근이 두 번 필요하다는 단점이 있다. 데이터에 접근하려면 페이지 테이블 항목을 읽기 위한 메모리
접근(1)과 실제 데이터를 읽기 위한 메모리 접근(2), 총 두 번의 메모리 접근이 필요하기 때문이다.

이 비효율적인 방식 떄문에 TLB(Translation Look-aside Buffer) 캐시 메모리를 사용한다.

## TLB (Translation Lookaside Buffer)

TLB는 CPU 내부에 있는 소규모 고속 캐시로, 최근에 사용된 페이지 번호와 프레임 번호의 쌍을 저장한다. 페이지 테이블 전체를
메모리에서 조회하는 대신 TLB를 먼저 확인하여 메모리 접근 횟수를 줄인다.

- TLB hit : 목표로 하는 페이지가 TLB에 있는 경우
- TLB miss : 묙표 페이지가 TLB에 없는 경우

#### EAT (Effective Access Time, 유효 메모리 접근 시간)

Hit Ratio 에 따라 Effective Memory-Access Time이 달라진다. Hit Ratio 는 TLB에서 페이지 번호를 찾을 확률이다. 메모리 접근 시간을 n이라고 할 때 EAT는 다음과 같이
계산된다.

```text
EAT = Hit Ratio * n + (1 - Hit Ratio) * (n * 2)
```

TLB Miss일 때 *2를 해주는 이유는 페이지 테이블 접근과 실제 데이터 접근, 두 번의 메모리 접근이 필요하기 때문이다.

## 페이징의 메모리 보호 방법

Protection Bit를 사용하여 메모리 영역을 보호한다.  
페이지 테이블의 각 항목에는 valid-invalid 비트가 추가로 붙어있다. 이 비트가 valid로 설정되면 해당 페이지가 프로세스의 논리 주소 공간에 속하는 합법적인 페이지임을 의미하고, invalid로
설정되면해당 페이지가 프로세스의 논리 주소 공간에 속하지 않는 불법적인 페이지임을 의미한다. 불법적인 주소에 접근하면 valid-invalid 비트에 의해 트랩(Trap)이 발생한다.

트랩이 발생하면 CPU가 현재 실행 중인 프로세스를 중단하고 운영체제로 제어권을 넘기고, 운영체제는 Segmentation Fault 오류를 발생시켜 강제 종료시킨다.

## Shared Pages : 공유 페이지

**Reentrant Code(재진입 가능 코드)** 를 여러 프로세스가 공유하여 메모리를 효율적으로 사용할 수 있다. 물리 메모리에 하나의 복사본만 올려두고 각 프로세스의 페이지 테이블이 동일한 프레임을 가리키는
방식으로 사용한다.  
재진입 가능 코드란 실행 중에 자신이 바뀌지 않는 코드로 읽기 전용 코드를 의미하며, 읽기 전용이므로 여러 프로세스가 동시에 접근해도 동기화 문제는 발생하지 않는다.

## Page Table의 구조화

논리 주소 공간이 커질수록 페이지 테이블 자체의 크기도 커지는 문제가 발생한다.
이를 해결하기 위 페이지 테이블을 계층적으로 나누는 Hierarchical Paging, 해시 함수를 이용하는 Hashed Page Table, 그리고 프레임 기준으로 테이블을 구성하는 Inverted Page
Table이다.

### Hierarchical Paging

페이징 테이블(A)을 여러 개의 나누고 그 페이징을 관리하는 별도의 페이징 테이블(B)를 만드는 방식이다.
논리 주소 공간을 여러 단계의 페이지 테이블로 나누는 방법이다.

### Hashed Page Table

주소 공간이 32비트 보다 큰 경우 사용한다. 가상 페이지 번호를 해시 값으로 변환하여 테이블을 관리하는 방식으로, Hash를 사용하므로 효율적인 탐색이 가능하다.

### Inverted Page Table

페이지 번호를 관리하는 방식이 아니라, PID(프로세스 ID)를 사용하여 어떤 프로세스가 어떤 페이지를 가지고 있는지 역으로 관리하는 효율적인 방법이다.

## Swapping : 스와핑

스와핑(Swapping)은 여러 프로세스가 사용하는 물리 주소 공간의 합이 실제 물리 메모리를 초과할 때도 프로그램을 실행할 수 있도록 하는 기법이다. 예를 들어 물리 메모리가 4GB인데 여러 프로세스가 합산
5GB를 필요로 할 때, 당장 필요한 부분만 메모리에 적재하고 나머지는 디스크의 백킹 스토어(Backing Store)에 보관한다. 메모리에서 백킹 스토어로 내보내는 것을 `Page Out`, 백킹 스토어에서
메모리로 불러오는 것을 `Page In`이라 한다. 이 동작을 합쳐서 `Swapping`이라고 한다.

> \*백킹 스토어(Backing Store)  
> 스와핑에서 메모리에서 내보낸 프로세스를 임시로 저장하는 디스크 공간

프로세서 전체를 스와핑 하는 것을 Standard Swapping이라고 한다. 프로세스 전체를 스왑하는 비용이 너무 크기 때문에, 필요한 페이지만 스와핑을 진행한다. 현대 운영체제에서 페이징이라고 하면 일반적으로 이
방식을 의미하며, 가상 메모리와 함께 사용할 때 효과적으로 동작한다.

> \*가상 메모리(Virtual Memory) 는 실제 물리 메모리보다 더 큰 메모리를 사용할 수 있는 것처럼 보이게 하는 기법이다.
