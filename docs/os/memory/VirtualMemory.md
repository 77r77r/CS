## Virtual Memory

가상 메모리(Virtual Memory)는 실제 메모리를 늘리는 것이 아니라 늘어난 것처럼 '보이게'하는 것으로, 논리 주소 공간을 물리 메모리보다 크게 보이게 만드는 기법이다. 프로세스가 메모리에 다 올라와 있지
않아도 실행할 수 있고 물리 메모리보다 큰 프로그램도 실행 시킬 수 있다.

> 구현 기법이 페이징이다.

주 기억장치(Main Memory)를 매우 큰 저장 공간으로 추상화하여 물리 메모리와 논리 메모리를 분리할 수 있다.

> 메인 메모리를 추상화한다는 것은 ‘실제 RAM 위에 가상적인 주소 공간을 얹어, 프로세스에게 논리 메모리라는 가짜(추상) 메모리를 보여주는 것’이다.

파일이나 라이브러리를 공유하거나 프로세스를 생성할 때 효율적이다.

필요할 떄마다 메모리 맵(Memory Map)을 통해 물리 메모리를 찾아간다. 실제 로딩되는 페이지는 물리 메모리에, 나머지는 서브 메모리(백킹 스토어)에 저장하고 있다가 필요한 경우에 스와핑을 통해 불러온다.

## Virtual Address Space

가상 주소 공간(Virtual Address Space)은 프로세스가 메모리에 어떻게 저장되어 있는지를 나타내는 논리적인 구조이다. 보통 0부터 시작하여, text, data, heap ~ stack 영역까지 연속된
메모리 공간에 존재하는 것처럼 보인다.

## Virtual Memory

파일과 메모리의 공유를 효율적으로 할 수 있다. 페이지 공유(Shared Pages)를 통해 Shared Memory가 가능하기 때문이다.

## Demand Paging

요구 페이징(Demand Paging)은 실제 필요한 페이지만 올리는 방법

전체 프로그램을 다 올릴 필요가 없으니 페이지를 잘라서 올리자. 그러면 언제 올리냐? 요청(Demand)할 때 올리자.
요청할 때만 올리면 많은 문제가 생겨서 이걸 해결해야줘야한
가상 메모레 있는 demand-paged 를 어떻게 관리할 건가? 실행중에 실제로 필요해지면(demanded)에서 메모리에 가정됨

메모리에 있는지 보조 메모리에 있는지는 하드웨어가 valid-invalid bit(유효 비트)로 판단을 한다.
valid : 페이지가 유효하고 메모리에 로딩되어 있음, invalid - 유효하지 않거나, 세컨드리 스토리지에 저장되어 있음.
또는 더티비트를 사용하여 별도로 구분하는 경우도 있다.

### Page Fault

메모리에 로징되지 않은 페이지에 접근하려고 하면 Page Fault가 발생한다.
페이지 부재가 발생하면 아래와 같이 동작한다.

1. 내부 페이지 테이블을 확인하여 유효한 메모리 접근인지 잘못된 메모리 접근인지 판단한다.
2. 유효한 접근이지만 해당 페이지가 메모리에 없다면 page fault(페이지 부재)가 발생한다. (잘못된 접근이면 프로세스를 종료한다.)
3. CPU는 trap을 발생시켜 운영체제(OS)로 제어를 넘긴다.
4. 보조 저장 장치에서 페이지를 읽어온다.
5. free-frame list에서 비어있는 프레임을 찾아 페이지를 할당한다.
6. 내부 페이지 터이블에 해당 페이지의 유효 비트를 valid로 변경한다.(reset page table)
7. 이후 명령어를 실행한다.

```text
CPU가 페이지 접근
      ↓
페이지가 메모리에 없음(page fault 발생)
      ↓
trap 발생 (OS 호출)
      ↓
OS가 디스크에서 페이지 로드
      ↓
페이지 테이블 수정
      ↓
중단된 명령 다시 실행
```

> 페이지 폴트가 발생하면 CPU는 trap을 발생시켜 운영체제로 제어를 넘긴다. 해당 프로세스는 I/O 요청이 완료될 떄까지 wait queue로 이동된다. 디스크에서 페이지 로딩이 완료되면 I/O
> interrupt가 발생하고, 운영체제는 해당 프로세스를 ready queue로 이동시킨다. 이후 스케줄러가 해당 프로세스를 선택하면 다시 running 상태에서 실행이 재개된다.

#### Pure Demand Paging

요청이 있을 때만 페이지를 메모리에 가져오는 방식이다. 프로그램이 시작되어도 페이지를 미리 로드하지 않고, Page Fault가 발생할 때만 해당 페이지를 메모리에 적재한다. 그때그때 페이지를 로드하는 방식이라 초기
실행 시 Page Fault가 자주 발생해 시간이 걸린다는 단점이 있다.

### 참조 지역성(Locality of Reference)

참조 지역성(Locality of Reference)은 프로그램이 특정 메모리 영역을 집중적으로 반복 접근하는 특성을 의미한다.

만약 프로그램이 계속 새로운 페이지에 접근한다면 페이지 폴트가 자주 발생할 수 있다.
그러나 실제 프로그램은 보통 자주 사용하는 코드와 데이터가 제한된 영역에 집중되어 있기 때문에, 한 번 메모리에 올라온 페이지를 반복해서 사용하게 된다.

이처럼 데이터 구조와 프로그램 구조를 적절히 설계하면 참조 지역성이 증가하고, 그 결과 페이지 폴트가 감소하여 시스템 성능이 향상된다.

### Hardware Support

요구 페이지에는 하드웨어의 지원이 필수적인다.

1. **Page Table**은 가상 페이지와 연결된 물리 페이지를 알려주고, valid-invalid 비트를 사용하여 각 페이지가 현재 메인 메모리에 존재하는지 여부를 알려준다.
2. **보조 저장장치**가 필요하다. 메인 메모리에 올라와 있지 않은 페이지를 저장하기 위한 공간으로 보통 swap space라고 한다. 일반적으로 고속 디스크나 NVM 장치로 구성된다.
3. **Instruction Restart**란, 페이지 폴트 이후 중단된 명령어를 동일한 상태에서 다시 실행하는 것이다. 페이지 폴트가 발생하면 OS는 현재 실행중인 프로세스의 상태를 저장한다. 이때 레지스터
   값, 조건 코드, PC 등 실행 상태 정보가 함께 저장된다. 이 정보를 통해 프로세스를 중단되었던 정확한 위치와 동일한 상태에서 실행할 수 있다.
    - 페이지 폴트의 발생 위치가
        1. 명령어를 가져오는(fetch) 중에 발생했다면, 해당 명령어를 다시 가져오면서 재시작한다.
        2. 피연산자(operand)를 가져오는 발생했다면, 먼저 명령어를 다시 가져와 해석(decode)한 뒤, 이후 피연산자를 다시 가져오는 과정을 수행한다.
   > ex) `ADD A, B, C` 인경우 ADD는 연산자, A, B, C는 피연산자이다.
4. **Free Frame List**는 페이지를 적재할 때 사용할 수 있는 빈 프레임들의 목록이다. 보조 저장장치에서 가져온 페이지를 메모리에 적재해야 할 때 이 목록에서 프레임을 하나 선택하여 사용한다. 또한,
   프로세스의 **스택(stack)**이나 힙(heap) 영역이 확장될 때도 필요하다. 이 경우에도 프리 프레임 리스트에서 빈 프레임을 할당하여 프로세스의 메모리 공간을 확장한다.

### 퍼포먼스에 영향을 미치는 요소들

EAT(effective access time)에서 메모리 엑세스 타임이 ma일때 페이지 폴트가 일어날 확률은 p 일 때 아래 수식을 사용한다.

- ` EAT = (1 − 𝑝) × 𝑚𝑎 + 𝑝 ∗ (𝑝𝑎𝑔𝑒 𝑓𝑎𝑢𝑙𝑡 𝑡𝑖𝑚𝑒) `

page fault time이란 보조 저장장치에서 페이지를 가져오는 시간이다. 인터럽트(trap)를 걸어주는 시간, 페이지를 읽어오는 시간, 프로세스를 재시작하는 시간을 포함하지만, 대부분 페이지를 읽어오는 시간에
크게 영향을 받는다.

### Copy-on-Write

프소세스가 공유 페이지에 쓰기(Write)를 수행할 때에만 페이지를 복사하는 기법이다. 이 방식은 불필요한 메모리 복사를 줄여 메모리 사용량과 프로세스 생성 비용을 줄이는 효과가 있다.

자식 프로세스가 부모를 복사할 때(fork), 동일한 논리 주소를 할당한다. (동일한 논리 주소, 동일한 물리 주소)

- 겉모습: 부모의 0x100번지에 "A"가 있다면, 자식의 0x100번지에도 "A"가 있는 것처럼 보인다.
- 실제(물리 메모리): 하지만 운영체제는 실제 RAM에 데이터를 두 번 복사하지 않고, 자식의 설계도(0x100)가 부모가 쓰던 실제 RAM의 그 위치를 똑같이 가리키게 슬쩍 연결만 해둔 것이다.

이후 부모나 자식 중 하나가 페이지에 쓰기(write) 작업을 수행하면 새로운 물리 주소를 할당한다. 즉 논리 주소는 동일하지만 물리 주소는 달라지게 된다.

> 각 프로세스가 가진 실행 상태 정보(CPU 상태와 주소 공간)가 서로 독립적이기 때문에 가능하다.

### Page Replacement

페이지 교체(Page Replacement)는 메모리에 빈 프레임이 없을 때, 현재 사용되지 않는 페이지를 디스크로 내보내고 그 자리를 새 페이지가 사용하도록 하는 것이다. 이때 어떤 페이지를 교체할지 선택하는 것이
페이지 교체 알고리즘이다.

페이지 폴트가 발생하면 운영체제는 먼저 **보조 저장장치(secondary storage)** 에서 필요한 페이지의 위치를 찾는다. 이후 해당 페이지를 적재할 **빈 프레임(free frame)** 이 있는지
확인한다. 만약 빈 프레임이 존재하면 그 프레임을 바로 사용한다.

빈 프레임이 없다면 **페이지 교체 알고리즘(page-replacement algorithm)** 을 사용하여 교체할 victim frame을 선택한다. 선택된 프레임의 페이지가 수정된 상태라면 그 내용을 **보조
저장장치에 기록(page out)** 하고, 페이지 테이블과 프레임 테이블을 갱신하여 해당 페이지가 더 이상 메모리에 존재하지 않음을 표시(invalid)한다.

그 후 페이지 폴트를 발생시킨 필요한 페이지를 디스크에서 읽어와(page in) 새롭게 확보된 프레임에 적재하고, 페이지 테이블과 프레임 테이블을 다시 갱신한다. 마지막으로 페이지 폴트가 발생했던 지점부터 프로세스
실행을 재시작한다.

### Demand Paging의 문제

Demand Paging을 구현할 때 운영체제는 두 가지 중요한 문제를 해결해야 한다.

첫 번째는 **프레임 할당 알고리즘(frame-allocation algorithm)** 이다. 각 프로세스에게 물리 메모리의 프레임을 몇 개씩 할당할 것인지를 결정하는 문제이다. 프로세스마다 필요한 메모리 요구량이
다르기 때문에 프레임을 어떻게 분배하느냐에 따라 시스템 성능이 크게 달라질 수 있다.

두 번째는 **페이지 교체 알고리즘(page-replacement algorithm)** 이다. 이는 메모리에 빈 프레임이 없을 때 어떤 페이지를 메모리에서 제거하고 새로운 페이지로 교체할 것인지를 결정하는
문제이다.

특히 보조 저장장치(디스크)와의 I/O 비용은 매우 크기 때문에, 페이지 폴트가 자주 발생하면 시스템 성능이 크게 저하된다. 따라서 Demand Paging에서 프레임 할당 방식이나 페이지 교체 방식이 조금만
개선되더라도 전체 시스템 성능에 큰 향상을 가져올 수 있다.

### Page Replacement : 페이지 교체 알고리즘

#### FIFO Page Replacement

FIFO Page Replacement는 가장 먼저 메모리에 들어온 순서대로 페이지를 교체하는 알고리즘이다. 페이지가 메모리에 들어온 순서로만 교체 대상이 결정된다. 일반적으로 이를 구현하기 위해 큐(queue)
구조를 사용한다.

1. 새로운 페이지가 메모리에 들어오면 큐의 뒤에 추가된다.
2. 페이지 교체가 필요하면 큐의 앞에 있는 가장 오래된 페이지를 제거한다.
3. 그 자리에 새로운 페이지를 로드한다.

이 방식은 구현이 매우 단순하지만, 최근에 자주 사용되는 페이지도 오래 전에 들어왔다는 이유만으로 교체될 수 있다는 문제가 있다. 그 결과 페이지 폴트가 많이 발생할 수 있으며, 대표적으로 **Belady’s
anomaly**가 발생할 수 있다.

> Belady’s anomaly?  
> 할당된 프레임 수를 증가했는데 페이지 폴트(page fault)가 오히려 증가하는 현상이다.

#### Optimal Page Replacement(OPT)

최적 교체 알고리즘(Optimal Page Replacement)는 **페이지 교체 알고리즘의 기준(reference)** 이 되는 알고리즘이다.
앞으로 가장 오랫동안 다시 사용되지 않을 페이지를 교체하는 방식으로 페이지 폴트의 비율이 제일 낮은 알고리즘이다. 하지만 페이지의 교체 순서를 알 수 없기 때문에 다른 알고리즘과 효율을 비교하는 기주으로 사용된다.

#### LRU Page Replacement

SJF와 비슷한 맥락으로 과거를 보고 미래를 예측하는 방식에서 착안한 것이 **LRU(Least Recently Used)** 알고리즘이다. 각 페이지가 마지막으로 사용된 시점을 기록하고, 가장 오랫동안 사용되지
않은 페이지를 교체한다. 성능이 좋아 실제 운영체제에서 자주 사용되며, Optimal 알고리즘과 마찬가지로 Belady's Anomaly가 발생하지 않는다.

구현 방식은 크게 두 가지다.

**Counter 방식**은 페이지를 참조할 때마다 타임스탬프를 기록하고, 교체 대상을 선정할 때 가장 오래된 값을 가진 페이지를 선택한다. 보통 프레임 테이블에 타임스탬프를 함께 저장하는 형태로 구현한다.

**Stack 방식**은 페이지 번호를 스택 구조로 관리하며, 페이지가 참조될 때마다 해당 페이지를 스택의 맨 위로 이동시킨다. 스택의 가장 아래에 있는 페이지가 교체 대상이 된다. 스택 중간 항목을 제거해야 하는
특성상 실제로는 **이중 연결 리스트(Doubly Linked List)** 로 구현한다.

두 방식 모두 구현이 복잡하고 추가적인 하드웨어 지원이 필요하다. 그래서 실제로는 레퍼런스 비트를 활용한 **LRU 근사 알고리즘(LRU-Approximation)** 을 사용한다. 각 페이지에 **Reference
Bit**를 두어 참조 여부를 기록하는데, 초기값은 0이며 참조되면 1로 설정된다. 교체가 필요할 때 Reference Bit가 0인 페이지를 교체 대상으로 선택한다.

**Second-Chance 알고리즘**은 FIFO를 기반으로 Reference Bit를 함께 고려하는 방식이다. FIFO 순서로 교체 대상을 선택한 뒤 Reference Bit를 확인하여, 0이면 바로 교체하고
1이면 두 번째 기회를 부여한다. 두 번째 기회를 받은 페이지는 Reference Bit를 0으로 초기화하고 도착 시간을 현재 시점으로 갱신하여 큐의 뒤쪽으로 이동시킨다.

### Frame Allocation

프레임 할당 문제는 여러 프로세스가 존재할 때 물리 메모리 프레임을 어떻게 분배할 것인지에 대한 문제이다.  
예를 들어 128개의 프레임이 있는 시스템에서 운영체제가 35개의 프레임을 사용하면 사용자 프로세스에 사용할 수 있는 프레임은 93개가 남는다.  
순수 요구 페이징 방식에서는 이 93개의 프레임이 free-frame list에 저장되며 초기 93번의 페이지 폴트는 이 프레임을 할당받아 해결된다. 그러나 94번째 페이지 폴트부터는 페이지 교체가 발생한다. 이때
시스템에 여러 프로세스가 존재하면 남은 93개의 프레임을 각 프로세스에 어떻게 나누어 줄 것인지가 중요한 문제가 된다.

> N개의 프로세스가 존재할 때 M개의 프레임이 존재하면 각 프로세스당 몇 개의 프레임을 할당할 것인가?

#### 프로세스의 크기에 따라 할당하는 방식

1. **Equal allocation(동등 할당)**은 모든 프로세스에 동일한 수의 프레임을 할당하는 방식이다.
2. **Proportional allocation(비례 할당)**은 프로세스의 크기에 비례하여 프레임을 할당하는 방식이다.

#### 페이지 교체 범위에 따라 할당하는 방식

1. **Local replacement**는 해당 프로세스에 할당된 프레임 내에서만 교체할 페이지를 선택하는 방식이다.
2. **Global replacement**는 시스템 전체 프레임 중에서 교체할 프레임을 선택하는 방식이다.

전체 시스템 성능을 높이기 위해서는 `Proportional Allocation + Global Replacement` 조합을 사용하고, 프로세스 간 간섭을 줄이기 위해서는
`Proportional Allocation() + Local Replacement` 방식을 사용한다.

### Thrashing

프로세스에 충분한 프레임이 할당되지 않으면 Page Fault가 매우 자주 발생한다. 그 결과 필요한 페이지를 계속 디스크에서 메모리로 가져오고 다시 내보내는 작업에 대부분의 시간을 소비하게 되는데, 이 상태를
Thrashing이라고 한다. 실제 프로그램 실행보다 페이지 교체 작업이 더 많이 발생하여 시스템 성능이 크게 저하된다.

### Degree of Multiprogramming

Degree of Multiprogramming은 메모리에 동시에 올라와 있는 프로세스의 수를 의미한다. 이 값이 증가하면 CPU 활용률은 처음에는 함께 증가하지만, 일정 수준을 넘어서면 각 프로세스에 할당되는
프레임이 부족해져 Page Fault가 급격히 증가한다. 결국 시스템이 페이지 교체와 디스크 I/O에 대부분의 시간을 소비하게 되면서 CPU 활용률이 급격히 떨어지는 Thrashing이 발생한다.

### Working-Set Model

Working-Set Model은 프로그램이 특정 메모리 영역을 집중적으로 사용하는 참조 지역성(Locality) 특성을 기반으로 한 메모리 관리 모델이다.
Δ(델타) 크기의 Working-Set Window 를 정의하고, 최근 Δ번의 페이지 참조에서 사용된 페이지들의 집합을 Working Set 으로 간주한다.
현재 활발히 사용되는 페이지는 Working Set에 포함되며, 일정 시간 동안 참조되지 않은 페이지는 Working Set에서 제거된다.
