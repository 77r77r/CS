## Deadlock : 교착상태

대기 중인 스레드(프로세스)가 자신의 상태를 변경할 수 없는 상태를 말한다. 스레드가 필요한 자원이 다른 대기중인 스레드에게 점유되어 있기 때문에 발생한다. 한정된 자원을 가진 시스템에서 여러 스레드가 경쟁할 때
나타나며, 같은 종류의 자원이라면 어떤 객체인지는 중요하지 않다.

스레드는 자원을 Request(요청) -> Use(사용) -> Release(반납) 순서로 사용하며, 사용 단계의 임계 구역에서 여러 자원이동시에 사용될 때 데드락이 발생할 수 있다.

### 데드락 발생의 4가지 필요조건

데드락은 아래 4가지 조건이 동시에 만족될 때 반드시 발생한다. 하나라도 만족되지 않으면 발생하지 않는다.

1. Mutual Exclusion : 상호 배제  
   한 번에 하나의 프로세스만 공유 자원을 사용할 수 있어야 한다.

2. Hold & Wait : 점유와 대기  
   자원을 점유한 스레드가 추가 자원을 얻기 위해 대기해야 한다.

3. Non-Preemptive : 비선점  
   다른 프로세스에 할당된 자원을 강제로 뺏을 수 없어야 한다.

4. Circular Wait : 환형 대기  
   프로세스들이 원형으로 구성되어, 각자 다음 프로세스가 점유한 자원을 기다려야 한다.

### 데드락 해결 방법

1. 무시하고 문제가 발생하면 해별하는 방법 : 무시(Ignorance)
2. 데드락이 발생하지 않도록 사전에 예방하는 방법 : 예방(Prevention)
3. 데드락을 회피하도록 구성하는 방법 : 회피(Avoidance)
4. 데드락이 발생한 후 탐지하여 복구하는방법 : 검출 및 복구(Detection & Recovery)

### Deadlock Prevention : 사전 예방

4가지 필요조건 중 하나 이상이 발생하지 않도록 막는 방법이다.

1. 상호배제가 발생하지 않는 경우는 데드락을 고려할 필요가 없으므로 신경쓰지 않는다.
2. 점유와 대기는 다른 자원을 홀딩하고 있는 것이 문제이므로, 자원을 추가로 요청할 때 기존에 점유한 자원을 모두 반납하고 다시 할당받도록 한다. 하지만 실제로 적용하기 어렵다.
3. 비선점 조건을 깨려면 자원을 강제로 빼앗는 방법을 사용할 수 있다. 빼앗긴 자원은 대기열로 들어가 기존에 대기 중이던 스레드에 할당될 수 있어 주의가 필요하다.
4. 환형 대기 를 막는 것이 현실적으로 가장 적용하기 쉬운 방법이다. 자원에 순서(번호)를 부여하고, 자원을 요청할 때 현재 점유 중인 자원보다 번호가 높은 것만 요청하도록 강제한다. 데드락은 막을 수 있지만
   Starvation(기아) 문제가 발생할 수 있다.

> 멀티스레딩의 장점을 데드락 예방 로직이 상쇄해버리는 문제가 있어, 실무에서는 데드락 회피 기법을 주로 사용한다.

### Deadlock Avoidance : 회피

자원을 요청할 때 데드락이 발생하지 않을 것 같으면 허용하고, 발생할 것 같으면 거절하는 방법이다. 아예 데드락 상태에 진입하지 않도록 하는 것이 핵심 아이디어다. 이를 위해 각 스레드가 앞으로 요구할 최대 자원
수를 미리 알고 있어야 한다.

**Safe State : 안전 상태**  
데드락이 발생하지 않는 상태로 시스템이 항상 안전 상태를 유지하면 데드락을 회피할 수 있다.  
핵심은 데드락 없이 모든 프로세스를 순서대로 실행시킬 수 있는 Safe Sequence를 찾는 것이다.

- **Single Instance**  
  자원 종류당 인스턴스가 1개인 경우 → Resource-Allocation Graph 알고리즘 사용
- **Multiple Instance**  
  자원 종류당 인스턴스가 여러 개인 경우 → Banker's Algorithm 사용

> \* Banker's Algorithm  
> 은행원 알고리즘은 Safe State를 유지하기 위한 자원 할당 알고리즘  
> 시스템이 안전하게 처리할 수 있는 범위 내에서만 자원을 할당

Safety Algorithm

현재 상태가 Safe State인지 확인하는 알고리즘이다. Safe Sequence를 찾을 수 있으면 Safe State, 없으면 Unsafe State다.

### Deadlock Detection

데드락 발생을 허용하되, 주기적으로 탐지 알고리즘을 실행해 데드락을 발견하면 복구하는 방법이다.  
탐지 알고리즘을 실행하는 빈도는 데드락의 발생 빈도와 스레드의 수에 따라 달라진다.

탐지 후 복구 방법은 두 가지다.  
- **Process Termination** : 프로세스 종료  
데드락에 걸린 프로세스를 강제 종료하는 방법으로 데드락에 걸린 프로세스를 전부 종료하거나, 데드락이 풀릴 때까지 하나씩 종료하며 확인한다.
- **Resource Preemption** : 자원 선점  
데드락에 걸린 프로세스에서 자원을 강제로 빼앗아 다른 프로세스에 할당하는 방법이다.  

- **Single Instance** : Wait-for Graph를 사용  
Resource-Allocation Graph에서 자원 노드를 제거하고 스레드 간의 대기 관계만 남긴 그래프로, 사이클이 존재하면 데드락이 발생한 것
- **Multiple Instance** : Safety Algorithm과 유사한 방식으로 탐지
